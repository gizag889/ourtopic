# Semantic Chunking & Note Text RAG - Implementation Plan

本計画では、これまでのXの投稿を用いたRAGプロセスに加え、ユーザーが手動でペーストした「長文（Note記事など）」を分析対象とするための開発を行います。長文を扱うため、意味合いの切れ目で文章を分割する「セマンティック・チャンキング（Semantic Chunking）」アルゴリズムを導入します。

## 1. アプリケーションの全体方針（フロントエンド）

現在の検索窓に対し、**タブ切り替え**（例:「Xの投稿を検索」 / 「長文を分析(ペースト)」）等のUIを導入します。

- **新規入力UI**:
  - トピック名（分析テーマ）の入力枠。
  - テキストペースト用の大きな `<textarea>`。
- **UI表示の分岐**:
  - 現在の `AxisCard.tsx` は X のツイートを表示するために `<Tweet id={...} />` コンポーネントを固定で使用しています。
  - 今回の機能では、表示するべき実体が「Note記事の一部（テキストの切り出し）」となるため、ツイートのIDが存在しません。
  - そのため、分析結果の表示部分を改修し、「Twitterモードならツイートを表示」「テキストモードなら抽出されたチャンク（テキスト）を引用枠（Blockquote）で表示」するように出し分けます。

## 2. APIエンドポイントと分析ロジック設計（バックエンド）

重い処理ロジックを分離するため、既存の `/api/analyze` とは別に、新しいエンドポイント **`/api/analyze-text/route.ts`** を作成するか、あるいはペイロードの `mode` プロパティによって処理を分岐させます。（今回は処理をスッキリ保つため、同じエンドポイント内で `mode` によって処理を分岐させるアプローチを提案します。※別のファイルにする方が良ければご指示ください）

### Semantic Chunking アルゴリズムの詳細フロー

ユーザーが指定したロジックに基づき、以下のアルゴリズムで意味的チャンキングを実施します。

1. **Sentence Segmentation (文分割)**
   - `Intl.Segmenter('ja-JP', { granularity: 'sentence' })` を用いて、長文を「。」「！」「？」などの単位の配列に分割します。
2. **Embedding (ベクトル化)**
   - Gemini Embedding API を用いて、全ての「文」のベクトルを一括で生成（EmbedMany）します。
3. **Similarity Calculation (類似度計算) と Boundary Detection (境界検出)**
   - 文 $i$ と 文 $i+1$ のコサイン類似度を比較します。
   - **境界の判定基準**: 
     - 類似度が設定した絶対的しきい値（例：`0.7` 以下）を下回った場合。
     - あるいは、その谷が前後の文と比べて十分に「深い（極小値である）」場合。
4. **Chunking (結合)**
   - 検出された境界で文リストを切断し、各チャンク内の文を文字列として結合します。
   - こうして出来上がった複数のチャンク（パラグラフ）を、疑似的な「ユーザーからの投稿群（意見リスト）」として扱い、既存のGemini生成（Opinion Topography抽出）に流し込みます。

### 代表意見（メドイド）の算出について

作成した「チャンク化されたテキスト群」について、これまでのツイートと同様に「各対立軸の極に最も近いチャンク」と「その極側の重心（メドイド）に最も近いチャンク」を算出して返却するようにします。

## 3. User Review Required (要確認事項)

> [!CAUTION]
> **1. APIのレスポンス型の変更**
> 現在 `AxisCard` へのデータとして `representative_tweets` というIDの配列を返していますが、テキストモードではツイートIDがありません。そのため、生のテキストを返す `representative_texts` および `medoid_texts` プロパティを追加するか、`source_type` を追加して型を統合する必要があります。この変更に伴い、UIの描画コンポーネントを条件分岐させます。

> [!WARNING]
> **2. チャンキングのしきい値**
> 「類似度 0.7」を最初の基本値として設定しますが、文章の性質（ブログか、論文か）によって分割点が細かすぎたり荒すぎたりする可能性があります。運用しながらしきい値を調整（または UI 側でユーザーがスライダーで調整できるようにするなどのはかり）が必要になるかもしれません。ここではまず固定値（内部ハードコード）で実装を進めてよろしいでしょうか。

> [!IMPORTANT]
> **3. エンドポイントの分離**
> 既存の `app/api/analyze/route.ts` にロジックを相乗りさせるか、それとも `app/api/analyze-text/route.ts` として全く新しいルートを作成するか。保守性の観点から「別ファイル（新しいルート）」として作成する方針で進めたいと考えますが、よろしいでしょうか？

## 4. 実行プラン
1. 承認が得られ次第、UIにタブ機能（Twitter検索・テキストペースト）を追加。
2. `AxisCard.tsx` の表示コンポーネントを、Tweetと生テキストの両対応に拡張。
3. `app/api/analyze-text/route.ts` を新規作成し、Semantic Chunkingと抽出ロジックを含んだ新分析パイプラインを実装。
4. 連携テストと検証を実施。
